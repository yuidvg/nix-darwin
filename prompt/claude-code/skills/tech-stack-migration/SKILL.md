## Phase 1: Extract — 現行アプリの完全な仕様抽出

現行アプリのソースコード、設定ファイル、DB定義、UIを網羅的に分析し、以下の5つのドキュメントを生成せよ。
各ドキュメントは **それ単体で現行アプリの該当側面を完全に再構築できるレベルの詳細度** を持つこと。

### 1. `Features.md` — 機能仕様書

以下を網羅すること：

- **機能一覧表**: 機能名 / 概要 / 対応画面 / 対応API / 優先度(P0-P3)
- **ユーザーストーリー**: 各機能のAs-a / I-want / So-that形式
- **ビジネスロジック**: バリデーションルール、計算ロジック、状態遷移図
- **認証・認可**: ロール定義、権限マトリクス、認証フロー
- **外部連携**: サードパーティAPI、Webhook、OAuth連携の一覧と仕様
- **非機能要件**: パフォーマンス目標値、同時接続数、レスポンスタイム上限

### 2. `Design.md` — デザイン仕様書

以下を網羅すること：

- **デザインシステム**: カラーパレット（HEX値）、タイポグラフィ（フォント名・サイズ・ウェイト）、スペーシング体系、角丸・シャドウ等のトークン一覧
- **コンポーネントカタログ**: 再利用コンポーネント一覧（ボタン、フォーム、モーダル、トースト等）と各状態（default / hover / active / disabled / error）
- **レスポンシブ戦略**: ブレークポイント定義、各ブレークポイントでのレイアウト変化ルール
- **アニメーション・トランジション**: 使用箇所、duration、easing関数
- **アクセシビリティ**: ARIA属性の使用パターン、キーボード操作対応、コントラスト比

### 3. `Pages.md` — 画面仕様書

以下を網羅すること：

- **画面一覧表**: パス / 画面名 / 認証要否 / 主要機能 / 対応APIエンドポイント
- **画面遷移図**: Mermaid形式のステートダイアグラム
- **各画面の詳細仕様**:
  - ワイヤーフレーム（テキスト形式 or ASCII）
  - 表示要素と配置
  - インタラクション一覧（クリック / 入力 / スクロール等のイベントと結果）
  - 初期表示時のデータ取得フロー
  - エラー状態・ローディング状態・空状態の表示仕様
- **共有レイアウト**: ヘッダー / サイドバー / フッター等のシェルコンポーネント仕様

### 4. `API.md` — API仕様書

以下を網羅すること：

- **エンドポイント一覧表**: メソッド / パス / 概要 / 認証要否 / レート制限
- **各エンドポイントの詳細**:
  - リクエスト: ヘッダー、パラメータ、ボディのスキーマ（型・必須/任意・バリデーション制約）
  - レスポンス: 各ステータスコード別のボディスキーマ
  - エラーレスポンス: エラーコード体系と各エラーの条件
  - リクエスト/レスポンスの具体例（JSON）
- **認証方式**: トークン形式、リフレッシュ戦略、セッション管理
- **WebSocket / リアルタイム通信**: イベント名・ペイロード定義（該当する場合）
- **ミドルウェア / 共通処理**: CORS設定、ロギング、エラーハンドリングの共通パターン

### 5. `DataStructure.md` — データ構造仕様書

以下を網羅すること：

- **ER図**: Mermaid形式のEntity-Relationship Diagram
- **テーブル定義一覧**: テーブル名 / カラム名 / 型 / 制約 / デフォルト値 / インデックス
- **リレーション**: 外部キー関係、カスケード設定
- **マイグレーション履歴**: 現在のスキーマバージョンと主要な変遷
- **シード/初期データ**: マスターデータやEnum的なデータの定義
- **クライアントサイド状態**: グローバルステート構造（Redux / Zustand / Context等）、ローカルストレージ使用パターン
- **キャッシュ戦略**: どのデータがどのレイヤーでキャッシュされるか

## Phase 2: Plan — 移行計画の策定

Phase 1の成果物を入力として、以下の計画を策定せよ。

### 計画策定の原則

1. **最小デプロイ単位**: 各フェーズは独立してデプロイ・検証可能な最小単位とする
2. **リスク順序**: 技術的不確実性が高い部分を早期フェーズに配置する
3. **価値順序**: ユーザー影響が大きい機能を優先する
4. **並行可能性**: 同時進行可能なフェーズはチーム分けして並行実行する

### 各フェーズ（Phase 3〜N）の必須構成

各フェーズは以下の構造を持つこと。フェーズ数に上限はない（Phase 100を超えても良い）。できる限り細かく分割すること。

```markdown
## Phase X: [フェーズ名]

### 前提条件
- 完了済みであるべき先行フェーズ
- 必要な環境・ツール・アクセス権

### 達成条件（Definition of Done）
- このフェーズ完了時に実現されていなければならないこと（箇条書き）

### 担当チーム
- 担当: [Frontend / Backend / Infra / Data / QA]
- 並行可能: [Yes/No — 同時進行可能な他フェーズ]

### 実行計画
1. 具体的な作業ステップ（コマンド・ファイルパス・設定値レベルの粒度）
2. ...

### テスト計画
- **単体テスト**: 対象モジュール、テストケース概要
- **結合テスト**: 対象インターフェース、検証項目
- **回帰テスト**: 既存機能への影響確認範囲
- **手動テスト**: 自動化できない検証項目

### ロールバック手順
- このフェーズの変更を安全に巻き戻す手順
```

### 想定されるフェーズの大分類（参考）

以下は典型的な分類例であり、実際のアプリに合わせて調整せよ：

1. **基盤構築**: ターゲットTech Stackのプロジェクト初期化、CI/CD、開発環境
2. **データ層移行**: スキーマ定義、マイグレーション、データ移行スクリプト
3. **API層移行**: エンドポイントの再実装（1エンドポイント＝1フェーズが理想）
4. **フロントエンド移行**: 画面の再実装（1画面＝1フェーズが理想）
5. **インテグレーション**: 外部連携、認証、リアルタイム通信
6. **品質保証**: E2Eテスト、パフォーマンステスト、セキュリティテスト
7. **カットオーバー**: DNS切替、旧システム停止、監視設定

### チーム編成と並列実行戦略

Phase 2の計画策定時、以下を必ず行うこと：

#### 1. チーム定義

アプリの規模・複雑度に応じてチームを定義せよ。典型例：

| チーム | 責務 |
|--------|------|
| Team Infra | CI/CD、デプロイ基盤、環境構築、監視 |
| Team Backend | API層、ビジネスロジック、認証 |
| Team Frontend | 画面実装、コンポーネント、状態管理 |
| Team Data | スキーマ設計、マイグレーション、データ移行 |
| Team QA | テスト戦略、E2E、パフォーマンス検証 |

チーム数はアプリの実態に合わせて増減してよい。1チームで十分な場合は1チームでよい。

#### 2. 依存グラフとクリティカルパス

全フェーズの依存関係を **Mermaid Gantt チャート** で可視化せよ。

```markdown
gantt
    title Migration Schedule
    dateFormat  YYYY-MM-DD
    section Team Infra
    Phase 3 CI/CD基盤構築      :infra1, 2025-01-01, 3d
    Phase 7 本番環境構築        :infra2, after data2, 2d
    section Team Backend
    Phase 5 認証API移行         :back1, after infra1, 2d
    Phase 6 CRUD API移行        :back2, after infra1, 4d
    section Team Frontend
    Phase 8 共通コンポーネント  :front1, after infra1, 3d
    Phase 9 画面A実装           :front2, after front1, 2d
    section Team Data
    Phase 4 スキーマ移行        :data1, after infra1, 2d
    Phase 10 データ移行          :data2, after data1, 3d
```

- **クリティカルパス**（最長依存チェーン）を特定し、明示すること
- クリティカルパス上のフェーズには `[CRITICAL]` タグを付与すること

#### 3. 並列実行ルール

- **依存関係がないフェーズは必ず並列実行する** — 直列にしてはならない
- 各フェーズに「ブロッカー」（先行完了が必須のフェーズ）と「並走可能フェーズ」を明記すること
- チーム間で共有リソース（DB、API契約等）がある場合、**インターフェース契約を先行フェーズで確定**し、各チームはモック/スタブで並行開発を進めること

#### 4. 同期ポイント（Sync Milestones）

並列チームが合流する同期ポイントを明示的に定義せよ。各同期ポイントでは：

- 全チームの該当フェーズが完了していること
- 結合テストを実施すること
- 次の並列ブロックに進む承認を行うこと

典型的な同期ポイント：

| 同期ポイント | タイミング | 検証内容 |
|-------------|-----------|---------|
| SP-1: 基盤完成 | Infra + Data 基盤フェーズ完了後 | 開発環境でビルド・デプロイ・DB接続が通ること |
| SP-2: API結合 | Backend + Data API実装完了後 | 全エンドポイントがDBと結合して動作すること |
| SP-3: フルスタック結合 | Frontend + Backend 結合後 | 主要ユーザーフローがE2Eで通ること |
| SP-4: リリース判定 | 全フェーズ完了後 | 全テストパス、パフォーマンス基準達成 |

### Phase 2 完了条件（品質ゲート）

- [ ] 全フェーズが定義されている
- [ ] 全フェーズに達成条件・実行計画・テスト計画・ロールバック手順が含まれている
- [ ] フェーズ間の依存関係が明示されている
- [ ] Features.mdの全機能がいずれかのフェーズでカバーされている
- [ ] クリティカルパスが特定され、`[CRITICAL]` タグが付与されている
- [ ] Mermaid Ganttチャートで並列実行スケジュールが可視化されている

- [ ] 同期ポイントが定義され、各ポイントの検証内容が明記されている
- [ ] チーム間のインターフェース契約が、各チームの実装フェーズより前に確定するスケジュールになっている

## Phase 3〜N: Execute — 計画の実行

Phase 2で策定した計画を、フェーズ順に実行する。

### 各フェーズ実行時のプロトコル

1. **開始宣言**: 「Phase X を開始します。前提条件: [確認済み]」
2. **実行**: 実行計画に沿ってコードを書き、ファイルを作成する
3. **テスト実行**: テスト計画に沿ってテストを実行し、結果を報告する
4. **完了判定**: 達成条件を1つずつチェックし、全て満たされていることを確認する
5. **完了宣言**: 「Phase X 完了。達成条件: [全項目チェック済み]。次のPhaseに進みます。」

### 異常時の対応

- テスト失敗時: 原因を特定し、修正を試みる。3回試行しても解決しない場合はロールバック手順を実行し、問題を報告する
- 計画と実態の乖離: 計画の修正案を提示し、承認を得てから続行する

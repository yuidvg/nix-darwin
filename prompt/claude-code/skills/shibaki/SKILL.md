# しばき — 巨大タスク遂行プロトコル

## これは何か

AIエージェントが巨大なタスク（移行、リアーキテクチャ、ゼロからの構築等）を遂行する際の行動規範。
タスク固有のプロンプト（移行プロンプト等）と **組み合わせて** 使う。

---

## 原則

### 原則 0: 手を動かす前に全体を見通せ

**実装に1行でも触れる前に、必ず以下を完了せよ：**

1. ゴールの明文化 — 「何が出来上がれば完了か」を箇条書きで定義する
2. 現状の把握 — 今あるもの（コード、データ、環境）を全て調査し、書き出す
3. ギャップの特定 — ゴールと現状の差分を洗い出す
4. 作業の分解 — ギャップを埋めるための作業をフェーズに分割する

**この原則への違反は最も重い。** 「とりあえず書き始める」「やりながら考える」は禁止。

### 原則 1: フェーズに分けよ、限界まで細かく

- 1フェーズ = 1つの検証可能な成果物
- フェーズの粒度は「1フェーズが失敗してもロールバックで他に影響しない」が基準
- フェーズ数に上限はない。100を超えてよい
- 各フェーズには必ず **達成条件** と **テスト計画** を定義する

### 原則 2: テストなき実装は未完了である

- コードを書いたらテストを書く。テストが通ったら初めて「完了」
- フェーズ完了時に達成条件を1つずつチェックし、全項目パスを宣言する
- 「動いているように見える」は完了ではない。テストが証明したものだけが完了

### 原則 3: 並列化せよ、直列に甘えるな

依存関係がないフェーズを直列に並べることは **怠慢** である。

- 依存グラフを描き、並列実行可能なフェーズを特定する
- チームに分けて同時進行する（チーム定義は後述）
- チーム間の接合面（API契約、データスキーマ等）を先に確定し、モック/スタブで並行開発する
- 同期ポイントで合流し、結合テストを行う

### 原則 4: マネージャーは手を動かすな

巨大タスクでは、**計画・監視・判断** を行うマネージャーロールと、**実装** を行うチームロールを分離する。

マネージャーの責務：
- フェーズ進捗の追跡
- チーム間の依存関係の調整
- 異常時の判断（リトライ / ロールバック / 計画変更）
- 品質ゲートの判定

**マネージャーが実装に手を出し始めたら、管理が崩壊しているサイン。**

### 原則 5: 異常は即報告、3回で撤退

- テスト失敗・想定外の事態が起きたら、まず状況を報告する
- 同じ問題に対するリトライは最大3回。3回失敗したらロールバックし、原因と代替案を報告する
- 「もう少しやれば直る」という希望的観測で突っ走らない

## 実行モデル: マネージャー + チーム

単一のAIエージェントが巨大タスクを遂行する場合、以下のように振る舞う：

```
┌─────────────────────────────────────────────┐
│  Manager（自分自身の上位モード）              │
│  - 全体計画の策定と管理                       │
│  - フェーズ完了判定                           │
│  - チーム間調整                               │
│                                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │ Team A   │ │ Team B   │ │ Team C   │      │
│  │ Backend  │ │ Frontend │ │ Infra    │      │
│  │          │ │          │ │          │      │
│  │ Phase 5  │ │ Phase 8  │ │ Phase 7  │ ← 並列 │
│  │ Phase 6  │ │ Phase 9  │ │          │      │
│  └──────────┘ └──────────┘ └──────────┘      │
│                                               │
│  ──── Sync Point: 結合テスト ────             │
│                                               │
│  ┌──────────┐ ┌──────────┐                    │
│  │ Team A   │ │ Team QA  │                    │
│  │ Phase 11 │ │ Phase 12 │ ← 次の並列ブロック │
│  └──────────┘ └──────────┘                    │
└─────────────────────────────────────────────┘
```

### マネージャーモードの行動パターン

```
[Manager] Phase 2 計画策定を開始。

== 計画出力 ==
（Ganttチャート、依存グラフ、チーム割当）

[Manager] 並列ブロック1を開始。Team Backend: Phase 5-6、Team Frontend: Phase 8-9、Team Infra: Phase 7。

[Team Backend] Phase 5 を開始。前提条件: Phase 3 完了済み ✓
（実装・テスト）
[Team Backend] Phase 5 完了。達成条件: 全項目 ✓

[Team Frontend] Phase 8 を開始。前提条件: Phase 3 完了済み ✓
（実装・テスト）
[Team Frontend] Phase 8 完了。達成条件: 全項目 ✓

[Team Infra] Phase 7 を開始。
（実装・テスト）
[Team Infra] Phase 7 完了。達成条件: 全項目 ✓

[Manager] 同期ポイント SP-2。全チーム完了。結合テストを実施。
（結合テスト実行）
[Manager] SP-2 パス。並列ブロック2に進行。
```

### チームへの切替時の規律

チームモードで実装中は、**そのフェーズのスコープだけに集中する。**

- 他フェーズの問題に気づいても、今のフェーズを完了してからマネージャーに報告する
- 「ついでにこれも直しておこう」は禁止。スコープ外の変更はマネージャーが別フェーズとして計画する
- 1フェーズの実装中に計画全体を見直したくなったら、マネージャーモードに戻ってから行う

## フェーズ設計テンプレート

```markdown
## Phase X: [フェーズ名] [CRITICAL]?

**担当**: Team Backend
**ブロッカー**: Phase 3, Phase 4
**並走可能**: Phase 8, Phase 9
**推定規模**: S / M / L / XL

### 達成条件（Definition of Done）
- [ ] 条件1
- [ ] 条件2

### 実行計画
1. ステップ1（コマンド・ファイルパスレベルの粒度）
2. ステップ2

### テスト計画
- 単体: ...
- 結合: ...
- 回帰: ...

### ロールバック手順
1. ...
```

## 進捗報告フォーマット

各同期ポイントまたはユーザーからの進捗確認時に、以下の形式で報告する：

```
=== 進捗報告 ===
完了: Phase 3, 4, 5, 7, 8  (5/15)
進行中: Phase 6 (Team Backend), Phase 9 (Team Frontend)
未着手: Phase 10-15
ブロック中: なし
クリティカルパス遅延: なし
次の同期ポイント: SP-2（Phase 6, 9 完了後）
========================
```

## アンチパターン（これをやったら失敗する）

| やりがちなこと | なぜダメか | 正しい行動 |
|--------------|-----------|-----------|
| 計画を書かずにコードを書き始める | 全体像が見えないまま局所最適に陥る | 原則0を徹底する |
| 「テストは後で」 | 後で書かれたテストはない | 実装とテストはセットで1フェーズ |
| 全部直列に並べる | 3倍の時間がかかる | 依存グラフを描いて並列化 |
| 1フェーズが巨大 | 失敗時の影響が大きく、ロールバックが困難 | 検証可能な最小単位まで分割 |
| マネージャーが実装を始める | 全体の見通しが失われる | 管理と実装のモードを明確に分ける |
| エラーを握りつぶして先に進む | 後で爆発する | 即報告、3回リトライ、撤退 |
| 「ついでにこれも」 | スコープクリープで品質崩壊 | スコープ外は別フェーズとして計画 |

## 使い方

タスク固有のプロンプトの冒頭に、以下のように参照する：

```
あなたは「しばきプロトコル」に従って行動してください。
{{しばきプロトコル全文、またはファイル参照}}

以下のタスクを遂行してください：
{{タスク固有のプロンプト（移行プロンプト等）}}
```

# Collect and Reply — 返信の収集・統合・返信作成

複数の情報源（人からの返信、スクショ、テキストメッセージ、メール）から得られた情報を
構造化して整理し、元の相談者に対する返信を作成する。

## トリガー

### 自動トリガー

- `/ask-network` で送ったメッセージに対する返信が集まり始めたとき
- ユーザーが「返信が来た」「これで返事を書きたい」と言ったとき
- スクリーンショットや転送メッセージが複数共有されたとき

### 手動トリガー

- `/collect-and-reply`
- 「まとめて返信して」「集まった情報で返事書いて」

## 前提

- 元の相談（誰から何を聞かれているか）のコンテキストが必要
- 返信元の人物が複数いて、それぞれ異なる情報を持っている
- 最終的に元の相談者に1通の構造化された返信を送る

## 実行手順

### Phase 1: 元の相談コンテキストの確認

以下を把握する（会話の流れで自然に。足りない情報だけ聞く）:

1. **元の相談者**: 誰から、どんな質問が来ているか
2. **元の質問の構造**: 質問が複数あるなら、それぞれを整理
3. **既存のやり取り**: 既にメールやメッセージを送っているか → Gmail スレッドを確認
4. **返信の言語**: 元の相談者の言語に合わせる（デフォルト: 元のやり取りの言語）

**Gmail 確認**: 元の相談者とのメールスレッドがある場合、最新の状態を確認して「何を約束したか」「相手はどう返したか」を把握する。

### Phase 2: 返信情報の収集・分類

ユーザーから共有される情報（スクショ、テキスト、メール転送など）を受け取り、以下のカテゴリに分類する:

| カテゴリ | 説明 | 例 |
|---|---|---|
| **具体的リード** | 人名、組織名、論文、URLなど直接使える情報 | 「大野太郎（財総研）がマイクロシミュレーションの論文を出している」 |
| **現実的な制約** | 「難しい」「存在しない」「まだ整備されていない」系の情報 | 「税データは少ない」「法律はコード化されていない」 |
| **取り次ぎ中** | 「聞いてみる」「返事しておく」 | 「すぐに名前は出てこないけど、聞いてみますね」 |
| **ネガティブ情報** | 明確に「ない」「いない」 | 「デジ庁もこども家庭庁もそこまで至ってない」 |

### Phase 3: 情報の統合・構造化

分類した情報を、元の質問の構造に対応する形でまとめる。

#### 統合の原則

1. **元の質問に対応させる**: 元の相談者が4つ質問していたら、4つのセクションで答える
2. **有用度でランク付け**: 最も具体的なリードを最初に、ネガティブ情報は「現実的な制約」として後半に
3. **複数人からの一致した情報は強調**: 「複数の人から同じフィードバックがあった」は説得力を増す
4. **情報源は匿名化**: 具体的に「誰が言った」は書かない（相手のネットワーク内の人間関係を出さない）。ただし、紹介可能な人名・組織名・論文は明示する
5. **ネガティブ情報も価値がある**: 「この分野は日本ではまだ初期段階」は重要な情報

#### 出力構造

```
1. [最も有用なリード — 人名、論文、組織など]
2. [次に有用なリード]
3. [関連する政府機関・研究所]
4. [現実的な制約 — 複数人からの一致したフィードバック]
まとめ — best starting points を簡潔に
```

### Phase 4: 返信メッセージの作成

#### 作成の原則

1. **元のやり取りの言語で書く**: 英語のスレッドなら英語、日本語なら日本語
2. **前回のメールを踏まえる**: 「Following up on my last email」のように、スレッドの文脈を引き継ぐ
3. **構造化して読みやすく**: 見出し・箇条書き・リンクを活用
4. **日本語の固有名詞は原語も併記**: 人名、組織名、論文タイトル（例: Taro Ohno (大野太郎)）
5. **正直に制約も伝える**: 良い情報だけでなく「ここは難しい」も含める
6. **次のアクションを示す**: 「紹介を仲介できる」「追加情報があれば送る」

#### 確認用の日本語版も作成

元の返信が英語の場合、ユーザー確認用に日本語訳版も同時に生成する。

### Phase 5: ファイル出力

**必ずファイルに書き出す**。

出力先: `~/Desktop/collect-and-reply-YYYY-MM-DD-TOPIC.md`

確認用日本語版がある場合: `~/Desktop/collect-and-reply-YYYY-MM-DD-TOPIC-ja.md`

出力後、両方のファイルを `open` コマンドで開く。

### Phase 6: 送信準備

ユーザーが内容を確認・修正した後:

1. **Gmail で下書き作成**: 既存スレッドへの返信として `Re: [元のSubject]` で下書きを作る
2. **ユーザーに Gmail リンクを提示**: 下書きの URL を渡して、Gmail 上で確認・送信してもらう

**注意**: Gmail API ではスレッドへの自動紐づけができない場合がある。その場合はユーザーに「Gmail 上でスレッド内から返信し、下書きの本文をコピペ」と案内する。

## /ask-network との連携

典型的なフロー:

```
/ask-network（前半）
  → 人脈を探す → メッセージを作成 → ユーザーが送信
  → [数日経過]
  → 返信が集まる
/collect-and-reply（後半）
  → 返信を収集 → 統合 → 元の相手に返信
```

ただし `/collect-and-reply` は単独でも使える:
- 自分で直接聞いた情報をまとめて誰かに返す
- 会議や雑談で得た情報を構造化してメールする
- 複数のソースからの調査結果を1通にまとめる

## 注意事項

- **情報提供者のプライバシー**: 「〇〇さんが言っていた」とは元の相談者に書かない。情報は匿名化して「I got feedback from multiple people」のように集約する。ただし紹介可能な人名（論文著者、公的な立場の人）は出してOK
- **待ち状態の判断**: まだ返信が来ていない人がいる場合、ユーザーに「待つか、今ある情報で返すか」を確認する
- **ネガティブ情報の扱い**: 「いない」「ない」という返答も重要な情報。「この分野はまだ初期段階」のようにフレーミングして伝える
- **ファイル出力必須**: ターミナルに直接出力しない
- **送信はユーザーが行う**: 下書きまで作成し、最終送信はユーザーに委ねる

## このスキルがやらないこと

- 人脈の探索やメッセージの作成（それは `/ask-network`）
- メッセージの実際の送信（下書き作成まで）
- 返信の催促や追加の問い合わせ（ユーザーが判断する）
